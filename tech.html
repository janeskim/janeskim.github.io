Ruby Recursions: Recognizing Patterns and Repetition


In Ruby, a recursion is a method that calls itself. Recursions are helpful because they allow us to break a problem down into smaller, repetitive parts. A recursive method will continue to call itself and execute these mini steps until the problem is finally solved.

Recursive methods have a few important characteristics. First, they include a base case. This is the condition that must be met in order for the recursion to stop. The recursion also specifies what happens to the code to reduce it to its end goal.

Here's a simple example of a recursion:



>> def counting(number)
>>   return if number == 5
>>   puts number
>>   countdown(number + 1)
>> end

>> counting(1)
1
2
3
4
5


In our example above, our recursive method takes one argument, number. Following this is our base conditional: return if number == 5. This means that our recursion will check the number each time, and if the number equals 5, the recursion will return the number 5 and exit the method. The method's main goal is to print out the number as indicated by puts number. Lastly, after the method has checked the number and printed it out, it calls itself inside itself countdown(number + 1) -- but this time, the method changes its argument by adding 1 to the number. That's end of our recursive method. Below that, we call the method and supply it with the argument counting(1). As you can see, our recursion continues until the number equals 5.

We could have used a much simpler loop to execute this action, so recursions aren't necessarily the best solution to every problem. The greatest difference between loops such as for, while and times loops is that recursions call the method within itself. A great example of the elegance and utility of a recursive method can be seen when calculating factorials.

Factorials multiply a series of descending natural numbers.
4! = 4 * 3 * 2 * 1

To write a formula for this, factorials essentially work like this:
n! = n * (n - 1) * (n - 2 ) * (n - 3) ... until n is equal to 1.

We're subtracting 1 from the new number 'n', so another way to look at factorials is like this:
n! = n * (n - 1)!

We also have our base conditional: the recursion stops when n is equal to 1.
Break loop when (n - 1) = 0

We have two repetitions, two small recursive loops in our factorial, and we also have our base conditional. Let's translate our idea into code:



>> def factorial(number)
>>   if number == 1
>>     return 1
>>   else
>>     return number * factorial(number - 1)
>>   end
>> end

>> factorial(4)
24


We have all the ingredients to our recursion above: using a simple if/else statement, we created the base condtional that will break the loop and return 1 when the number is equal to 1. Otherwise, our recursion calls itself with a small modification to the argument return number * factorial(number - 1) so that we can reduce our recursive method to its final solution. When we call our method factorial(4), it returns 24.

Recursions are complex loops that provide a lot of elegance to a solution when recursions are applicable. Here are some great resources on recursions:
-The Bastards Book of Ruby
-The Odin Project

JavaScript Objects: Understanding Data Structures in JavaScript


When it comes to understanding JavaScript, it's important to note that nearly everything in JavaScript is an object. This means that variables and data are objects as well. Much like Ruby, there are various ways we can organize data using variables in JavaScript.

A JavaScript object is defined as an unordered collection of variables called 'named values'. That's a whole lot of technical jargon, so let's break it down. JavaScript objects contain data, and this data is organized in a specific way: instead of numbering the data, objects keep track of the data (also called values) by giving a name to the data -- hence, named values. The JavaScript term for the value's name is property. In other words, properties are like variables that refer to a value.

To think about objects in JavaScript more concretely, let's make a simple comparison to a real life object: a house. We want to create a specific house that has one story, 2 bedrooms, 1 bathroom, and a red roof. The story, bedroom, bathroom, and roof are all properties of the object and the numbers and colors are its values. Here's how we can express this in JavaScript:



>> var house = {
>>   story: 1,
>>   bedroom: 2,
>>   bathroom: 1,
>>   roof: "red"
>> }



In the example above, we use the JavaScript keyword var to relay to the computer that we're establishing a variable. Next, we set this variable to a set of curly brackets { } which contains our data. The property is connected to its value using a colon : and each named value pair is separated by a comma. We can use numbers or strings to express values.

Wait a minute... a named data structure with curly brackets that holds a set of paired data separated by commas -- sound familiar? JavaScript objects are similar to Ruby hashes, which organize data using key/value pairs. If you'd like a refresher on Ruby data structures, you can check out my previous blogpost here.

Blocks, Procs and Lambdas: More on Object Oriented Programming


BLOCKS

A Ruby block is any code that is executed using the syntax do and end or curly braces { }. In a previous blog post, we used blocks to iterate over arrays and hashes and blocks are also used to execute loops. Here are some examples of blocks:



>> [10, 20, 30].each { |number|
>>   puts number * 2
>> }
20, 40, 60

>> 3.times do
>>   puts "This block will print itself out three times."
>> end
"This block will print itself out three times."
"This block will print itself out three times."
"This block will print itself out three times."



We've been working with blocks for a while, and now we know how to officially identify them. Awesome! Now let's extend our knowledge of blocks and move into procs.


PROCS

Procs are saved blocks. When we create a block, we have the option of assigning it to a proc. This allows us to reuse this block without having to type it out all over again. In other words, procs allow us to name a block, call that name using specific syntax, and execute the code inside the block. Procs are useful because they help maintain important Ruby rule: Don't Repeat Yourself.

Here's the syntax for creating a proc using Proc.new. In our example below, we're converting integers to strings using the method #to_s and returning a destructive array using #collect! and #map!. You can call a proc inside parentheses using the symbol &.



>> return_string = Proc.new { |number| number.to_s }

>> [1, 2, 3].collect!(&return_string)
["1", "2", "3"]

>> [4, 5, 6].map!(&return_string)
["4", "5", "6"]



LAMBDAS

Lambdas are similar to procs in that you can designate a block of code and reuse it. When it comes to writing and calling a lambda, the syntax is very similar to a proc. Check out the example below:



>> name_of_lambda = lambda { |parameter| block }

>> double = lambda { |number| number * 2 }

>> [1,2,3].map!(&double)
[2,4,6]

>> [4,5,6].collect!(&double)
[8,10,12]



LAMBDAS VS. PROCS

Lambdas and procs are similar in many ways but they have a few important differences. First off, a lambda checks the number of arguments that are passed to it whereas a proc does not. In the instance where a wrong number of arguments are passed, a lambda will return an error message while a proc will assign nil to its missing arguments. Another important difference between a proc and a lambda is the way they pass control inside of a method. When a lambda returns its value, it passes control to the calling method first. When a proc returns its value, it returns the value first without going back to the calling method. Here's an example explaining this in greater detail:



>> def calling_method_proc
>>   control = Proc.new { return "The proc goes first." }
>>   control.call
>>   "The calling method is returned first."
>> end

>> def calling_method_lambda
>>   pass = lambda { return "This is a lambda test." }
>>   pass.call
>>   "The calling method is returned first."
>> end

>> puts calling_method_proc
"The proc goes first."
>> puts calling_method_lambda
"The calling method is returned first."



With this new knowledge in mind, we can write Ruby code that is more concise with less repetition, code that is more DRY. I learned a lot about blocks, procs and lambdas using CodeAcademy's excellent tutorial.
-CodeAcademy on Blocks, Procs and Lambdas

Classes: Bringing Together Objects and Methods


Ruby is an object-oriented programming language. This means that getting anything done in Ruby requires working with objects: objects contain data and we ask objects to manipulate and return data in useful forms.

Last week we looked into an example of the method #each_by in the Enumerable module. All the methods in the Enumerable module only work on objects that are lists. This is because different types of objects can only execute certain types of tasks. An object's ability to use a method depends on its class.

Let's take a real life example: when we have the numbers "2" and "4" we can complete addition, subtraction, multiplication and division. When we have the words "cat" and "pig", we can't run those mathematical actions on the words. "cat" - "pig" = ? Doesn't work quite the same. If we think about this in Ruby terms, the objects "2" and "4" are in the class number and the objects "cat" and "pig" are in the class words. Different types of classes have certain characteristics and those classes can execute a specific set of methods.

In Ruby, we can actually create our own classes. We can also create objects that belong to a class and have the objects execute specific tasks. For our example, let's create a class of bikes. What are some of the characteristics a bike has? A bike has a color, wheels, and gears. What kinds of actions can a bike do? The bike can speed up, slow down and come to a full stop. Now, let's translate this into Ruby code.



>> class Bike

>>   def initialize(color, wheel_size, gears)
>>     @color = color
>>     @wheel_size = wheel_size
>>     @gears = gears
>>   end

>>   def specs
>>     puts "Your bike is #{@color}.
>>     puts "It's wheels are #{wheel_size} and it has #{@gears} speeds."
>>   end

>>   def ride
>>     puts "Pedal faster!"
>>   end

>>   def fixed
>>     new_gears = @gears - @gears
>>     puts "Your bike had #{@gears} gears but now it has #{new_gears}."
>>     puts "Whoops, we turned it into a fixed gear."
>>   end

>> end


This is a very simple example of how to create a new class. First, we declared our new class Bike by capitalizing the name of the class and putting class in front of it. Next, we defined a method called initialize. This method states that objects that belong to the class Bike must specify three characteristics: the color, the wheel size, and the gears. Whenever we declare an object under the class Bike, we have to supply that object with these characteristics.

We covered some of basic ways to establish the characteristics of our class, so let's move onto methods. Objects within a class can execute specific methods, and we create those methods inside the class definition. In our example, we defined three methods: specs, ride and fixed.

Let's see our new class in action.



>> turbo = Bike.new("red", "29 inches", 3)

>> turbo.specs
"Your bike is red."
"It's wheels are 29 inches and it has 3 speeds."

>> turbo.ride
"Pedal faster!"

>> turbo.fixed
"Your bike had 3 gears but now it has 0."
"Whoops, we turned it into a fixed gear."


Here we created a new object called turbo, defined it as belonging to the new class Bike and specified three characteristics. We called the methods specs, ride and fixed on the object turbo.

Ruby is a dynamic language and there are many ways to alter objects, add methods and change our classes, but the important thing to know is every object in Ruby belongs to a class and each class has a distinct set of characteristics and methods.

Enumerables: Iterating with Finer Control


Last week we discussed how arrays and hashes are ordered lists in this previous post. We touched upon methods and explained them briefly as commands that can be executed by objects. This time we're going more indepth into a related Ruby concept about methods called Enumerables.

The Enumerable is a module. A module is a collection of objects that share a common set of traits. In this case, all Enumerable share a specific structure: the object must be able to apply a method over elements within itself. For example, lists can apply a method over each item in the list. Arrays and hashes fall under the Enumerable module, and as a result, they can accept Enumerable methods. The repetition of a method over each item in an object is called iteration.

To talk more about how Enumerables iterate over an object, let's look at the Enumerable method, #group_by. Simply put, the method #group_by breaks a list into groups based on specified criteria. It takes a list as its input and returns a hash as its output.

Let's at the general syntax for #group_by then go into more specific examples.



>> input.group_by { |x| x }



Aside from a few exceptions, methods are called using a . period. In our syntax example, our input on the left side of the period is the object receiving the method. On the right side of the we have the method group_by which is being sent to the object.

Next are a pair of curly brackets. This is where we expand the method group_by with more specific information on how we want it to group the input. The variable within the pipes |x| acts as a placeholder for each item in the input list. Think of the pipes as a door: each item of the input collection is lining up to walk through these pipes in order access the block of code specified in the curly brackets. In our syntax example above, we only have x after the pipes, but we can expand this part of the code to specify how we want to group each of the items.

Let's take a look at a live example.



>> input_array = [ "Jane", "Jeff", "Tonya", "Hannah", "Harry" ]

>> input_array.group_by { |name| name.length }
{ 4 => ["Jane", "Jeff"] , 5 => ["Tonya", "Harry"] , 6 => ["Hannah"] }



We called the method .length on our variable name. The method counted the number of characters in each item in the input_array. The output hash took the length of item as the key and grouped the input into arrays for the value.

Here's another example.



>> input_array = [ 2, 6, 5, 4, 10, 3, 3, 51 ]

>> input_array.group_by { |number| number.even? }
{ true => [ 2, 6, 4, 10 ] , false => [ 5, 3, 3, 51 ] }



This time we used the method with a question mark at the end even? which returns a true or false, also known as a boolean value, and grouped the data accordingly.

These are the basics of Enumerables. Enumerable methods are important because they allow us to organize the data in arrays and hashes into useful and applicable forms. The Ruby-Docs are a foundational source for learning about methods, you can check out the documentation here:
-Ruby Docs for Enumerables

Data Structures: An Introduction to Arrays and Hashes


One of the fundamental building blocks of Ruby are variables. Just like algebra, we write Ruby variables to represent a specific value. We can also call a variable when we want to use its value in the code.



>> x = 1234567890
1234567890
>> x * 2
2469135780



In the example above, the variable x represents the number 1234567890. In Ruby, variables can hold numbers, strings (words and sentences) and boolean expressions (a term in Ruby referring to true or false statements).

Arrays and hashes are a specific type of variable called a data structure. Just like variables, data structures hold value but arrays and hashes are different in that they can hold more than one value. In other words, arrays and hashes are lists. The two data structures differ from each other in a few important ways, so let's examine the syntax and functionality of each item.


ARRAYS

This is how we write an array.



>> my_array = [1, 2, 3, 4, 5]



Just like the example of the variable x, we first give the array a name my_array and set it equal = to a value, which is the list contained within the brackets [1, 2, 3, 4, 5]. To note the obvious, we separate different items in the list using commas. This goes across the board for arrays and hashes. We can name our array anything we want and we can fill the array's brackets with any combination of numbers, strings and boolean expressions.

In order to access a specific value in the array, we have to know its index number. Naturally we would call 1 the first element in my_array, but arrays start from the index number 0. To print 1, this is how we would write it.



>> puts my_array[0]
1



In the example above, we use the method puts to print out 1. Methods are like commands, and arrays have a plethora of methods to manipulate the data. In our very first example with the variable x, we used the method * (or multiply) to manipulate our variable. As I mentioned before, there are so many methods we can use on the data in my_array, but the important thing to know is arrays are indexed lists.


HASHES

This is how we write a hash.



>> my_hash = { "name" => "Jane", "city" => "Chicago", "ice_cream" => "always"}



We named our hash my_hash and set it equal = to a set of values in the curly brackets { }. Values are separated by commas, just like arrays. While arrays refer to elements by the index number, hashes create key-value pairs. In the example for my_hash, the key "name" refers to its value pair "Jane"; the key "city" refers to its value pair "Chicago"; and the key "ice cream" refers to its value pair "always". Keys refer to their values using the symbol =>.

(I'm sure you also noticed all those quotation marks up there. We tell Ruby that we're using strings with quotation marks, but we don't need the quotation marks for numbers or booleans.)

Let's call the first key-value pair for my_hash. The code is pretty similar to arrays.



>> puts my_hash["name"]
"Jane"



Hashes are useful when we want to link two pieces of information together, such a list of clients and their phone numbers. There are many, many methods for accesing and using the data in hashes but the foundational take-away for hashes is they are key-value pairs.

Arrays and hashes are just a few examples of the basic ways we can organize data within Ruby. If you want more practice with the two, CodeAcademy has a helpful intro course.

CSS Positioning: The Difference between Relative, Absolute and Fixed Positioning


We use CSS to style our HTML elements such as changing the font on a paragraph or adding a color to the background, but we can also use CSS to place elements on a page.There are four types of CSS positioning values: static, relative, absolute and fixed. Let's take a look at an example. On the far left HTML column are four div elements; in the center is the CSS code styling those elements; and on the right are the elements rendered in their normal flow. The white div is the parent to the red and yellow div. The blue div rests below them in the normal flow.



When things are declared in the static positioning, they remain in the normal flow. Declaring the static position is the same as not declaring anything at all. HTML elements are rendered from top to bottom, and child and sibling elements sit inside the parent element when things are in static positioning. As you can see, we declared the static position and all the elements remain the same.



When we declare a relative position, elements remain in the normal flow but they are placed in relation to its position in the normal flow. By using the properties top, bottom, left, and right, we can move the element a certain number of pixels from the top, bottom, left or right of where it would normally be. Since the element is still part of the normal flow, it continues to work with the other HTML elements. Child elements will still be nested in their parent elements and elements will move from top to bottom, but we can specify values which will move the elements from their static position. In the example below, we declared a relative position for the red div and specified that it move 50px from the left.



Unlike static and relative positioning, absolute positioning takes an element completely out of the normal flow. Instead it is positioned relative to the browser window. We can use the properties top, bottom, left, and right to adjust the element's location on the page. Here we can see the red is taken out of the normal flow and positioned relative to the browser window, 90px from the bottom of it.



Similar to absolute positioning, the fixed position also takes the element out of the normal flow. However, it stays fixed to the position on the browser and does not scroll with the page. I'm sure you've seen websites with a navigation bar at the top that always remains there, even when you're scrolling through the page--that's an example of a fixed element.

Git, GitHub and the Command Line: Tools For Getting Started


This is my first week of Phase 0 at DBC and already I've learned so much! Unit 1 focused on the basics of git, GitHub, and the command line--all tools that were completely new to me at the start of this week.

THE COMMAND LINE

Let's start off by simply defining the command line. You know how you find files and folders on your Mac using Finder? The command line goes many layers underneath the visual functionality of Finder to execute similar tasks such as finding folders and files and installing software. The command line is an extremely powerful tool and I'm only hitting the surface of it, but it can essentially be described the heart of the computer.

GIT

To talk about git and GitHub, we first have to talk about version control. Version control is a software that keeps track of all the versions of a project and git is a version control software you install onto your computer using the command line. To understand how git works, imagine a tree: we have the main project, which is the trunk of our tree, called the master branch. In order to work on the project, git creates copies of the master branch and calls these copies branches. Since branches are copies of the project, they don't affect the master, and users can experiment within the branches to try out new ideas. When a branch contains a functionality that is awesome enough to be a part of the final product, we merge the branch onto the master using three steps: add, commit and push. As for the branches we don't want? We can simply delete those.

Because we add branches onto the master, git keeps track of all the changes made on a project through a collection of snapshots. The git log tracks all the versions of the project that came before, like a photo album of the work in progress. This is great because you can always travel back to a specific version--whether it be for debugging or experimenting--and work on the project at that particular state. Nothing is lost, all is carefully documented, and disaster is averted.

GITHUB

We've covered git, so what about GitHub? First off, it's important to note that GitHub is different from git. While git is a piece of software that exists locally on your computer, GitHub exists remotely on the internet. Although different, the two are not separate because git and GitHub work together. By syncing git on our computer to GitHub on the internet, we can send the work we do on our computer to GitHub using the git commands, add, commit and push. We also take projects from GitHub onto our computers, and these commands are called pull or fetch.

Here's a diagram to explain the git and GitHub workflow:


From Scott Chacon

Lastly, it's important to note git and GitHub are essentially collaborating tools and we must view the master branch as something untouchable and holy. It's the source code that many others are drawing from and adding to. As a result, there's usually a check/balance system where your pushed changes are reviewed by another team member before it's merged onto the master branch. In other words, when working on a project we can merge onto the master branch and we can draw from the master branch but we must never work on the master branch itself.
